# Fluent Bit systemd log streaming â€” design

**Status:** draft
**Date:** 2026-05-01
**Author:** unitctl

## Goal

Stream selected systemd journal logs from each unit to a central log server using
[Fluent Bit](https://docs.fluentbit.io/) over the **forward** protocol, optionally
secured with the same mutual-TLS material that MQTT already uses. Configuration is
generated by `unitctl` from `config.toml` at startup, mirroring the existing
`camera_env` / `mavlink_env` write-on-start pattern.

## Non-goals

- Auto-installing the Fluent Bit apt repository on non-Debian targets. The install
  script already targets Debian bookworm.
- Schema-validating `systemd_filter` keys against the canonical journald field set.
  Only structural validation (KEY=VALUE, no newlines).
- Multiple Fluent Bit outputs or pipelines beyond the single forward output.
- Buffer / disk-spool tuning. Defaults are sufficient for now.

## Architecture

A new `[fluentbit]` section in `config.toml` drives a `Task`-implementing env
writer (`FluentbitEnvWriter`) that emits a Fluent Bit YAML config at startup. A
new `fluentbit.service` runs `fluent-bit -c <config_path>`. A
`fluentbit-watcher.{path,service}` pair restarts it whenever the YAML is
regenerated â€” exactly mirroring the existing `camera` / `mavlink` precedent.

```
unitctl writes <config_path>  â†’  fluentbit-watcher.path fires
                              â†’  fluentbit-watcher.service: `systemctl restart fluentbit`
                              â†’  fluentbit.service tails journald â†’ forward â†’ central server
```

When `[fluentbit].enabled = false`, the writer skips file generation. The watcher
never fires and `fluentbit.service` stays inactive (its `ExecStartPre` blocks on
the file's existence). Flipping `enabled = true` and restarting `unitctl`
generates the file, the watcher fires, and Fluent Bit comes up.

## Config schema changes

### Move TLS cert paths to `[general]` (optional)

Both MQTT and Fluent Bit consume mutual-TLS material. To avoid duplicating the
fields and to keep validation simple, the three cert paths move from `[mqtt]` to
`[general]` and become optional:

```rust
pub struct GeneralConfig {
    pub debug: bool,
    pub interface: String,
    pub env_dir: String,
    pub ca_cert_path: Option<String>,
    pub client_cert_path: Option<String>,
    pub client_key_path: Option<String>,
}
```

`MqttConfig` loses these three fields. **Availability** is checked by each
consumer (`MqttTransport::new`, `FluentbitEnvWriter`) â€” `Config::validate()` only
checks string sanity (non-empty when `Some`, no `\n` / `\r`).

### New `[fluentbit]` section

```rust
pub struct FluentbitConfig {
    pub enabled: bool,
    pub host: String,
    pub port: u16,
    pub tls: bool,
    pub tls_verify: bool,
    pub config_path: String,                 // e.g. /etc/fluent-bit.yaml
    pub systemd_filter: Option<Vec<String>>, // each entry "KEY=VALUE"
}
```

`Config::validate()` checks (only when `fluentbit.enabled`):

- `host` non-empty and not `-`-prefixed.
- `port != 0`.
- `config_path` non-empty, no `\n` / `\r`.
- Each `systemd_filter` entry: contains exactly one `=`, no `\n` / `\r`, key part
  matches `[A-Z_][A-Z0-9_]*`.

`config.toml.example` excerpt:

```toml
[general]
debug = false
interface = "eth0"
env_dir = "/var/run/unitctl"
# Optional TLS material shared by MQTT and Fluent Bit (forward, when fluentbit.tls=true)
ca_cert_path = "/etc/unitctl/certs/ca.pem"
client_cert_path = "/etc/unitctl/certs/client.pem"
client_key_path = "/etc/unitctl/certs/client.key"

[fluentbit]
enabled = false
host = "logs.example.com"
port = 24224
tls = true
tls_verify = true
config_path = "/etc/fluent-bit.yaml"
# systemd_filter = ["_SYSTEMD_UNIT=unitctl.service", "_SYSTEMD_UNIT=mavlink.service"]
```

## `FluentbitEnvWriter` (`src/env/fluentbit_env.rs`)

Implements `Task`. On run:

1. If `!fluentbit.enabled`: log info, exit. (No file written â†’ watcher never
   fires â†’ service stays inactive.)
2. If `fluentbit.tls` and any of `general.{ca_cert_path, client_cert_path,
   client_key_path}` is `None` or empty: log error naming the missing field, do
   not write, exit.
3. Otherwise: render YAML and write to `config_path` (atomic
   write-then-rename), creating parent directories as needed.

A pure helper `generate_fluentbit_config(&Config) -> String` is unit-tested
directly, paralleling `generate_mavlink_env` / `generate_camera_env`.

### Generated YAML

```yaml
service:
  flush: 1
  log_level: info

pipeline:
  inputs:
    - name: systemd
      tag: host.*
      read_from_tail: off
      systemd_filter:
        - _SYSTEMD_UNIT=unitctl.service
        - _SYSTEMD_UNIT=mavlink.service

  outputs:
    - name: forward
      match: '*'
      host: logs.example.com
      port: 24224
      tls: on
      tls.verify: on
      tls.ca_file: /etc/unitctl/certs/ca.pem
      tls.crt_file: /etc/unitctl/certs/client.pem
      tls.key_file: /etc/unitctl/certs/client.key
```

Rendering rules:

- `systemd_filter` block omitted when the field is `None` or empty.
- `tls.*` block omitted entirely when `fluentbit.tls = false`. Cert paths are
  not required in that case.
- `tls.verify` is emitted as `on` / `off` based on `fluentbit.tls_verify`. Only
  emitted when `tls = true`.
- `read_from_tail` is hard-coded to `off` so Fluent Bit reads the journal from
  the cursor / start, not the live tail (boot-time logs are not lost).
- Dotted property names (`tls.verify`, `tls.ca_file`, â€¦) are valid in Fluent
  Bit's YAML parser and map to the same internal property names as the classic
  `.conf` format.

## `SafeConfig` redaction (`src/messages/commands.rs`)

- New `SafeGeneralConfig` mirrors `GeneralConfig` but replaces each `Some(path)`
  cert field with `Some("***")`.
- `SafeConfig.general: SafeGeneralConfig`.
- `SafeMqttConfig` shrinks (no cert fields).
- New `SafeConfig.fluentbit: FluentbitConfig` (no secrets in this section, so
  it can be cloned directly).
- `get_config` redaction tests assert redaction on `general.{ca,client}_*`
  and that `fluentbit` is exposed.

## `MqttTransport::new` validation

The constructor signature changes from `new(&MqttConfig, â€¦)` to `new(&Config,
â€¦)`. On construction, when `mqtt.enabled`:

- Require `general.ca_cert_path`, `general.client_cert_path`,
  `general.client_key_path` all `Some(non_empty)`. If any is missing, return a
  new `TransportError::MissingTlsConfig { field: &'static str }` naming the
  first missing field.
- The internal `tls::load_tls_config(...)` call now reads from `general.*`.
- All transport tests are updated to set the cert paths under `general`.

## systemd units (`services/`)

### `fluentbit.service`

```
[Unit]
Description=Fluent Bit log forwarder
After=unitctl.service network-online.target
Wants=network-online.target
StartLimitBurst=5
StartLimitIntervalSec=60

[Service]
Type=exec
TimeoutStartSec=30s
ExecStartPre=/bin/sh -c 'until [ -f /etc/fluent-bit.yaml ]; do sleep 0.1; done'
ExecStart=/opt/fluent-bit/bin/fluent-bit -c /etc/fluent-bit.yaml
Restart=on-failure
RestartSec=1s
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

The exact binary path (`/opt/fluent-bit/bin/fluent-bit` vs `/usr/bin/fluent-bit`)
is locked in during implementation, after confirming on Debian bookworm with the
official apt package.

### `fluentbit-watcher.path`

```
[Unit]
Description=Restart fluentbit on config changes
StartLimitIntervalSec=0

[Path]
PathModified=/etc/fluent-bit.yaml

[Install]
WantedBy=multi-user.target
```

### `fluentbit-watcher.service`

```
[Unit]
Description=Restart fluentbit on config changes
After=network.target
StartLimitIntervalSec=10
StartLimitBurst=20

[Service]
Type=oneshot
ExecStart=/usr/bin/systemctl restart fluentbit

[Install]
WantedBy=multi-user.target
```

## `scripts/install.sh`

### `install_packages()`

Add prerequisites and the Fluent Bit official apt repo, then install
`fluent-bit`:

- Install `curl` and `gnupg` (in addition to the existing list).
- Import the Fluent Bit signing key to `/usr/share/keyrings/fluentbit-keyring.gpg`.
- Write `/etc/apt/sources.list.d/fluent-bit.list` pinned to `bookworm`.
- `apt-get update && apt-get install -y fluent-bit`.

### `install()`

Mirror the camera/mavlink block:

```bash
echo "Setting up fluentbit services..."
systemctl-exists fluentbit.service || systemctl link ./services/fluentbit.service
systemctl-exists fluentbit-watcher.service || {
  systemctl link ./services/fluentbit-watcher.service
  systemctl enable ./services/fluentbit-watcher.path
  maybe_start fluentbit-watcher.path
}
```

### `uninstall()`

```bash
systemctl disable --now fluentbit-watcher.path || true
systemctl disable --now fluentbit.service || true
systemctl disable --now fluentbit-watcher.service || true
```

The composite `disable --now` line at the top of `uninstall()` also gains
`fluentbit-watcher.path`.

## Wiring in `main.rs`

After the existing env writers:

```rust
let fluentbit_env = Arc::new(FluentbitEnvWriter::new(Arc::clone(&ctx)));
handles.extend(fluentbit_env.run());
```

`MqttTransport::new` is updated to take `&ctx.config` instead of
`&ctx.config.mqtt`; the call site changes accordingly.

## Tests

### `generate_fluentbit_config` (unit)

`generate_fluentbit_config` is only called when `fluentbit.enabled = true`; the
disabled branch is exercised at the writer level (no file written).

- TLS on: includes `tls.ca_file` / `tls.crt_file` / `tls.key_file` and
  `tls.verify: on`.
- TLS off: omits the entire `tls.*` block; cert paths are not required.
- `systemd_filter = None`: key absent from rendered YAML.
- `systemd_filter = Some(vec![...])`: each entry rendered as a YAML list item
  under `systemd_filter:`.
- `read_from_tail: off` always present in the systemd input.
- Output is parseable by `serde_yaml` (round-trip sanity).

### `FluentbitEnvWriter` (integration)

- Writes file at the configured `config_path`.
- Creates parent directories.
- Disabled: no file written.
- TLS required but a cert path is missing: no file written; an error is logged
  (asserted via `tracing-test` or by capturing the writer's exit status).

### `Config::validate`

- Cert paths optional and accepted when `None`.
- Non-empty + no-newline checks fire when `Some`.
- `fluentbit.enabled` + empty `host` rejected.
- `fluentbit.enabled` + `port = 0` rejected.
- Malformed `systemd_filter` entries rejected (no `=`, newline, CR, lowercase
  key).
- Roundtrip via `toml::from_str` / `to_string`.

### `MqttTransport::new`

- Returns `MissingTlsConfig` when `mqtt.enabled` but `general.*` cert paths are
  absent.
- Existing happy-path tests pass after moving cert paths under `general`.

### `SafeConfig`

- Cert paths redacted under `general` (each `Some(_)` â†’ `Some("***")`).
- `mqtt` no longer carries cert fields.
- `fluentbit` present and unredacted (no secrets in that section).

## Migration / compatibility

This is a config-breaking change: any existing `config.toml` with
`mqtt.{ca,client}_*` will fail to parse. Callers must move the three keys to
`[general]`. Documented in `config.toml.example` and the section comments.

## Risks

- **Fluent Bit YAML availability.** YAML config has been stable in Fluent Bit
  â‰¥ 2.0. The official apt package on bookworm currently ships â‰¥ 3.x. If the
  pinned package version regresses to < 2.0, the writer must be retargeted to
  the classic `.conf` format. Implementation pins a known-good apt version.
- **Binary path drift.** `/opt/fluent-bit/bin/fluent-bit` (apt package) vs
  `/usr/bin/fluent-bit` (Debian package). Settled during implementation.
- **TLS handshake on a not-yet-routable network.** `fluentbit.service` waits on
  `network-online.target` to avoid this; the watcher path also re-triggers on
  every config regeneration so a transient startup failure is recovered as soon
  as `unitctl` rewrites the file.
