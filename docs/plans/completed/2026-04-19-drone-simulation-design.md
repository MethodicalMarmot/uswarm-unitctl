# Drone Simulation Environment — Design Spec

**Date:** 2026-04-19
**Status:** approved, ready for implementation planning
**Scope:** one-off feature landing three coordinated changes (Rust, shell, Docker) plus a new container image.

## 1. Goal

Produce a self-contained Docker image (`unitctl-sim:latest`) that runs the full
on-board drone stack with every external dependency simulated:

- **Flight controller** replaced by ArduPilot SITL (`arducopter`) emitting
  MAVLink over a native UART.
- **Camera** replaced by a synthetic gstreamer pipeline (`videotestsrc`)
  encoding H.264 and sending RTP to the configured GCS.
- **LTE modem** replaced by a `FakeModemAccess` implementation of the
  existing `ModemAccess` trait, answering the AT commands `LteSensor`
  issues.

The image runs real `systemd` as PID 1. `unitctl.service`, `sitl.service`,
`mavlink.service`, and `camera.service` all live inside the container.
Operators point the image at their MQTT credentials via a volume mount; no
other host configuration is required.

## 2. Non-goals

- No simulation of wfb-ng, RTK, or any radio-layer component.
- No production-image changes beyond the narrow config/trait additions
  needed for the simulation to exist as a compile-time configurable
  alternative.
- No docker-compose, Kubernetes manifests, or CI image publishing — a
  buildable `Dockerfile.sim` plus instructions is sufficient.

## 3. Architecture

### 3.1 Runtime process graph (inside the container)

```
PID 1  /sbin/init  (systemd)
  |
  |-- unitctl.service         [Type=simple]
  |     unitctl --config /etc/unitctl/config.toml
  |     writes /etc/mavlink.env and /etc/camera.env at startup
  |
  |-- sitl.service            [Type=simple, After=unitctl.service]
  |     ExecStartPre: socat PTY,link=/tmp/sim/ttyFC-sitl,raw,echo=0 \
  |                        PTY,link=/tmp/sim/ttyFC,raw,echo=0 &
  |     ExecStart:    arducopter --model quad -S \
  |                        --defaults=/opt/sim/copter.parm \
  |                        --uartA=uart:/tmp/sim/ttyFC-sitl
  |
  |-- mavlink.service         [Type=exec, After=unitctl.service sitl.service,
  |                            Requires=sitl.service]
  |     ExecStartPre: wait for /etc/mavlink.env
  |     ExecStart:    bash /opt/unitctl/scripts/run-mavlink.sh
  |                     (unchanged script; FC_TTY=/tmp/sim/ttyFC)
  |
  |-- camera.service          [Type=exec, After=unitctl.service]
        ExecStartPre: wait for /etc/camera.env
        ExecStart:    bash /opt/unitctl/scripts/run-camera.sh
                        (CAMERA_TYPE=fake routes through new script branch)
```

All four units are enabled under `multi-user.target`.

### 3.2 Serial bridge between SITL and mavlink-routerd

ArduPilot SITL's `--uartA=uart:<path>` flag opens `<path>` as a real serial
device; it does not create the device itself. We therefore create a PTY pair
up-front with `socat` (one invocation per container boot, backgrounded in
`sitl.service`'s `ExecStartPre`). The two symlinks — `/tmp/sim/ttyFC-sitl`
and `/tmp/sim/ttyFC` — are the two endpoints of the pair. SITL opens the
`-sitl` side via its native uart path; `mavlink-routerd` opens the
`/tmp/sim/ttyFC` side as `${FC_TTY}` from the env file. The `-sitl`
endpoint name is purely internal to `sitl.service`; `config.toml` only
needs to know about `/tmp/sim/ttyFC`.

Rationale for socat + native uart (rather than TCP bridge): keeps
`run-mavlink.sh` unchanged and exercises the same serial path as production,
which is the point of an integration simulator.

### 3.3 Configuration flow

Both services read env files generated by `unitctl` at startup
(`MavlinkEnvWriter`, `CameraEnvWriter`). A baked `config.toml` inside the
image sets:

| Key                      | Value in sim image          |
|--------------------------|-----------------------------|
| `mavlink.fc.tty`         | `/tmp/sim/ttyFC`            |
| `mavlink.fc.baudrate`    | `115200` (SITL default)     |
| `camera.camera_type`     | `"fake"`                    |
| `sensors.lte.modem_type` | `"fake"` (new field)        |
| everything else          | sensible simulator defaults |

Services' `ExecStartPre` polls for the env file with `until [ -f <path> ]; do sleep 0.1; done`
(capped with systemd's `TimeoutStartSec` so a crashed `unitctl.service`
doesn't hang the dependency chain indefinitely).

### 3.4 MQTT credentials

Operators pass MQTT certificates in via a read-only volume:

```
docker run --rm \
  --network host \
  -v /host/unitctl/certs:/etc/unitctl/certs:ro \
  --tmpfs /tmp --tmpfs /run \
  --cgroupns=host \
  unitctl-sim:latest
```

`--network host` is required so the container's MAVLink/video/MQTT traffic
can reach the host-configured `gcs_ip` and `mqtt.host` without additional
port-forwarding rules.

The baked `config.toml` references `/etc/unitctl/certs/ca.pem`,
`/etc/unitctl/certs/client.pem`, and `/etc/unitctl/certs/client.key` — the
standard paths the image expects. If the volume is omitted or empty, MQTT
simply fails to start (`unitctl` logs the failure and the rest of the stack
continues to run).

## 4. Rust changes

### 4.1 Config: `sensors.lte.modem_type`

Extend `LteSensorConfig` with a new required string field:

```rust
pub struct LteSensorConfig {
    pub enabled: bool,
    pub interval_s: Option<f64>,
    pub neighbor_expiry_s: f64,
    pub modem_type: String,          // "dbus" | "fake"
}
```

- Validated in `Config::validate`: value must be exactly `"dbus"` or
  `"fake"`.
- `config.toml.example` updated to set `modem_type = "dbus"`.
- JSON Schema regenerated.

### 4.2 Module split: `services/modem_access`

Move the existing single-file module to a directory module without changing
the public API consumed by `main.rs` and the sensors:

```
src/services/modem_access/
    mod.rs    pub trait ModemAccess, ModemAccessService,
              ModemType, ModemError, NetworkRegistration,
              discover_modem, send_at_command, detect_modem_type,
              parse_registration_response
    dbus.rs   DbusModemAccess (moved verbatim from the current inline
              `pub mod dbus` block)
    fake.rs   FakeModemAccess (new)
```

All existing tests stay in `mod.rs` (or a sibling `tests.rs`). No behavioral
changes to the D-Bus implementation.

### 4.3 `FakeModemAccess`

A deterministic, state-machine-driven implementation of `ModemAccess`:

- `model()` returns `"EM12"` so the existing Quectel EM12 parsing path in
  `LteSensor` exercises end-to-end.
- `command(cmd, _)` dispatches on a prefix match:
  - `AT+QENG="servingcell"` → a synthesized `+QENG` response with 20
    comma-separated fields whose numeric columns (RSRP, RSRQ, RSSI, RSSNR,
    tx_power, pcid, earfcn) are driven by a monotonic counter so the
    values drift smoothly. No randomness — tests must be deterministic.
  - `AT+QENG="neighbourcell"` → two neighbor cell lines (distinct PCIDs
    from the serving cell) with their own drifting values.
  - `AT+CIMI` → a fixed fake IMSI (e.g. `"001010123456789"`).
  - `AT+CREG?` / `AT+CEREG?` → `"+CEREG: 0,1\r\nOK"` (registered, home).
  - Anything else → `"OK"`.
- Internal counter guarded by a `Mutex<u64>` incremented per call (cheap,
  contention-free in practice because `ModemAccessService` already
  serializes calls through its worker).

Drift is engineered so the produced values are valid parse inputs for the
existing parsers — this is verified by dedicated round-trip tests
(`fake → parse_quectel_em12_serving → LteSignalQuality`).

### 4.4 `ModemAccessService::start` branching

New signature:

```rust
pub async fn start(
    cfg: &LteSensorConfig,
    cancel: &CancellationToken,
) -> Result<Arc<Self>, ModemError>
```

Behavior:

- `cfg.modem_type == "dbus"` → existing discovery-with-retry path.
- `cfg.modem_type == "fake"` → immediately wrap a `FakeModemAccess` in the
  worker-queue machinery (no discovery, no retries). Logs one info line so
  operators can tell sim mode is active.

Caller in `main.rs` updates accordingly:

```rust
ModemAccessService::start(&ctx.config.sensors.lte, &modem_cancel).await
```

### 4.5 Tests

- `FakeModemAccess` unit tests: each supported AT command returns a
  well-formed response that the existing parsers accept; unknown commands
  return `"OK"`.
- Integration-style test on `ModemAccessService`: with
  `modem_type = "fake"`, `start()` completes instantly (no D-Bus needed)
  and subsequent `command()` calls roundtrip through the worker.
- Config validation tests: `"dbus"` and `"fake"` accepted; `""`,
  `"DBUS"` (wrong case), and arbitrary strings rejected.

## 5. Shell script changes

### 5.1 `scripts/run-camera.sh` — new `fake` case

Add a new shell function and dispatch entry:

```bash
fake() {
  gst-launch-1.0 -q \
    videotestsrc is-live=true pattern=ball ! \
    video/x-raw,width=${CAMERA_WIDTH},height=${CAMERA_HEIGHT},framerate=${CAMERA_FRAMERATE}/1 ! \
    clockoverlay valignment=top halignment=left ! \
    timeoverlay valignment=bottom halignment=left ! \
    videoconvert ! \
    x264enc tune=zerolatency speed-preset=ultrafast \
            bitrate=$((CAMERA_BITRATE / 1000)) key-int-max=30 ! \
    rtph264pay config-interval=1 pt=96 mtu=1200 aggregate-mode=zero-latency ! \
    udpsink host=${GCS_IP} port=${REMOTE_VIDEO_PORT} sync=false
}

case "${CAMERA_TYPE}" in
  ...existing branches...
  "fake") fake ;;
  *) echo "Invalid camera type"; exit 1 ;;
esac
```

`CAMERA_DEVICE` is ignored in `fake` mode (the pipeline has no v4l2src).
The existing check `[ -z "${CAMERA_DEVICE}" ]` stays — operators must still
set it to a non-empty dummy value in config (documented).

### 5.2 `scripts/run-mavlink.sh`

No changes. It already handles a serial-style `FC_TTY` input.

## 6. Docker image

### 6.1 Dockerfile.sim — multi-stage

```dockerfile
# --- Stage 1: unitctl builder ---
FROM rust:1-bookworm AS unitctl-builder
WORKDIR /src
COPY . .
RUN cargo build --release --bin unitctl

# --- Stage 2: ArduPilot SITL builder ---
FROM debian:bookworm AS sitl-builder
RUN apt-get update && apt-get install -y --no-install-recommends \
    git python3 python3-pip python3-future python3-lxml \
    build-essential ccache g++ gawk make wget \
    ca-certificates
ARG ARDUPILOT_TAG=Copter-4.5.7
RUN git clone --depth=1 --branch=${ARDUPILOT_TAG} --recursive \
    https://github.com/ArduPilot/ardupilot.git /ardupilot
WORKDIR /ardupilot
RUN ./waf configure --board sitl
RUN ./waf copter

# --- Stage 3: runtime ---
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y --no-install-recommends \
    systemd systemd-sysv \
    socat bash ca-certificates \
    gstreamer1.0-tools gstreamer1.0-plugins-base \
    gstreamer1.0-plugins-good gstreamer1.0-plugins-bad \
    gstreamer1.0-plugins-ugly gstreamer1.0-libav \
 && rm -rf /var/lib/apt/lists/*

# Binaries
COPY --from=unitctl-builder /src/target/release/unitctl   /usr/local/bin/unitctl
COPY --from=sitl-builder    /ardupilot/build/sitl/bin/arducopter /usr/local/bin/arducopter

# Repo assets
COPY scripts/                       /opt/unitctl/scripts/
COPY mavlink-routerd/               /opt/unitctl/mavlink-routerd/
COPY sim/copter.parm                /opt/sim/copter.parm
COPY sim/config.toml                /etc/unitctl/config.toml
COPY sim/systemd/*.service          /etc/systemd/system/

# Enable services
RUN systemctl enable unitctl.service sitl.service mavlink.service camera.service \
 && mkdir -p /tmp/sim /etc/unitctl/certs

STOPSIGNAL SIGRTMIN+3
ENTRYPOINT ["/sbin/init"]
```

### 6.2 New image assets (new files committed to repo under `sim/`)

- `sim/Dockerfile.sim` — the multi-stage Dockerfile.
- `sim/config.toml` — the baked simulator config (sim values for all
  required fields).
- `sim/copter.parm` — ArduPilot parameter defaults for deterministic
  startup (copter standard + tweaks to accept SITL-native UART).
- `sim/systemd/unitctl.service`
- `sim/systemd/sitl.service`
- `sim/systemd/mavlink.service` — analogous to the existing
  `services/mavlink.service` but pointing at in-container paths and
  declaring `Requires=sitl.service`.
- `sim/systemd/camera.service` — analogous to `services/camera.service`
  with in-container paths.

### 6.3 Unit files (in-container)

All four unit files follow this pattern, varying only by ExecStart:

```ini
[Unit]
Description=<component>
After=unitctl.service  # plus sitl.service for mavlink.service
Requires=unitctl.service  # sitl.service requires unitctl.service

[Service]
Type=exec
TimeoutStartSec=30s
Restart=on-failure
RestartSec=1s
ExecStartPre=/bin/sh -c 'until [ -f /etc/<name>.env ]; do sleep 0.1; done'
EnvironmentFile=/etc/<name>.env
ExecStart=/bin/bash /opt/unitctl/scripts/run-<name>.sh

[Install]
WantedBy=multi-user.target
```

`unitctl.service` has no env-file wait and no `Requires`. `sitl.service`
has no env-file wait but uses `ExecStartPre` to launch socat in the
background and wait for the PTY symlinks to appear.

### 6.4 Host invocation

```
docker run --rm \
  --network host \
  -v /host/unitctl/certs:/etc/unitctl/certs:ro \
  --tmpfs /tmp --tmpfs /run --tmpfs /run/lock \
  --cgroupns=host \
  unitctl-sim:latest
```

(`--network host` makes the container's MAVLink/video/MQTT traffic reach
`gcs_ip` and `mqtt.host` as configured; `--cgroupns=host` is required for
systemd-in-docker on current Docker versions. Documented in the image
README.)

## 7. Validation

Following the existing repo conventions (`cargo test`, `cargo clippy`,
`cargo fmt --check`) plus:

1. `docker build -f sim/Dockerfile.sim -t unitctl-sim:test .` succeeds in a
   clean clone.
2. Container boots to `multi-user.target` within 30s without any service in
   `failed` state (`systemctl list-units --failed` is empty).
3. A host-side listener on `$GCS_IP:$REMOTE_MAVLINK_PORT` receives MAVLink
   heartbeats with `sysid=1`, `autopilot=ARDUPILOTMEGA` (SITL's signature).
4. A host-side listener on `$GCS_IP:$REMOTE_VIDEO_PORT` receives a valid
   H.264 RTP stream decodable by `ffmpeg -f rtp -i ...` for at least 10s.
5. `unitctl` logs a non-empty LTE reading (RSRP/RSRQ/RSSI within the ranges
   the fake modem produces) within 5s of start.
6. When `mqtt.enabled = true` and credentials are mounted, the node status
   transitions to `Online` and telemetry is published.

## 8. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Systemd-in-docker is version-sensitive (cgroup, tmpfs mounts). | Document the required `docker run` invocation in the repo README. Pin to a Debian bookworm base so the systemd version is stable. |
| ArduPilot build inflates image size / build time. | Multi-stage: only the compiled `arducopter` binary (~20 MB) is copied into the runtime stage. Builder layers are cached. |
| SITL's `--uartA=uart:<path>` requires the path to exist before the process starts. | socat `ExecStartPre` creates and backgrounds the PTY pair, with a small `until [ -e /tmp/sim/ttyFC-sitl ]` wait before exec'ing `arducopter`. |
| `FakeModemAccess` produces data that existing parsers reject. | Dedicated round-trip tests: generate → parse → assert parsed value is within expected drift bounds. |
| Baked config.toml + volume-mounted certs diverge from production config fields as schema evolves. | Generate the baked config from `config.toml.example` (sim variant) as part of the image build; any new required field causes the image build to fail validation. |

## 9. Out-of-scope follow-ups

- Publishing `unitctl-sim` to a registry and wiring it into CI.
- Adding additional SITL vehicle profiles (plane, rover).
- Exposing simulation controls (throttle, attitude injection) via an RPC
  endpoint.
- Refactoring the existing `services/mavlink.service` + `camera.service` on
  the host side to share more with the sim variants.
